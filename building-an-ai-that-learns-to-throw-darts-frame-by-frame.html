Building an AI That Learns to Throw Darts - Frame by Frame

Posted on Mon 09 March 2026 in Software • 6 min read

I wanted to make a short-form video where an AI visibly learns a physical skill from scratch - starting terrible, converging to competent. Darts felt perfect: there's a clear target, a throwing motion with real biomechanics, and enough parameters to make the "learning" look genuine. The result is a 9:16 reel rendered entirely in Node.js, no browser, no Puppeteer.

Here's how it actually works.

The rendering pipeline

The whole thing runs on node-canvas (a native binding to Cairo) piped into FFmpeg through stdin. No temporary frame files, no disk thrashing - just raw pixel buffers pushed directly to the encoder.

The backpressure handling is worth showing, because if you skip it on a machine with limited RAM you'll OOM halfway through:

async function writeAsync(proc, canvas) {
  const buf = canvas.toBuffer('raw');
  if (!proc.stdin.write(buf)) {
    await new Promise(r => proc.stdin.once('drain', r));
  }
}

At 720x1280 BGRA, each frame is ~3.5MB. At 24fps over a 60-second video, that's theoretically 5GB uncompressed if you tried to buffer it. The drain-and-wait pattern keeps memory flat. FFmpeg consumes frames as fast as it can encode them; we just wait when it asks us to slow down.

FFmpeg is spawned with stdin as a pipe and given the raw pixel format:

The canvas renders a frame, we call writeAsync, move to the next frame. Simple loop, zero file I/O.

Cross-Entropy Method for learning

The "AI" is a Cross-Entropy Method optimizer. CEM is one of those algorithms that's almost offensively simple once you understand it, but produces surprisingly good results for low-dimensional continuous problems.

The idea: maintain a Gaussian distribution over your parameter space. Sample N candidates, score them all, keep the top K (the "elite" set), and update the distribution mean toward the elite average. Shrink the variance a bit each generation. Repeat.

function cemTrain(scorer, initMu, initSig, minSig, opts = {}) {
  const { N = 10, elite = 3, decay = 0.97, smooth = 0.25 } = opts;
  const mu = { ...initMu }, sig = { ...initSig };
  const keys = Object.keys(mu);

  for (let gen = 1; gen <= 1000; gen++) {
    const samples = Array.from({ length: N }, () => {
      const p = Object.fromEntries(keys.map(k => [k, mu[k] + srn() * sig[k]]));
      return { p, score: scorer(p) };
    }).sort((a, b) => b.score - a.score);

    const elites = samples.slice(0, elite);
    keys.forEach(k => {
      const eliteMu = elites.reduce((s, e) => s + e.p[k], 0) / elite;
      mu[k] = (1 - smooth) * mu[k] + smooth * eliteMu;
      sig[k] = Math.max(minSig[k] || 0.01, sig[k] * decay);
    });
  }
}

The srn() is a seeded normal random - important for reproducibility, since the same training run needs to produce the same animation every time.

For the darts reel, three parameters are optimized simultaneously:

  • ang - the forearm angle at release (when to let go)
  • speed - throw force (initial dart velocity)
  • snapBias - wrist snap contribution, an angular offset added to the launch direction

They all start wrong. Release angle is 42 degrees too early. Speed is 26 when it should converge to around 18. Wrist snap is adding an extra 11 degrees of upward bias. Early generations throw the dart into the ceiling or the floor. By generation 100 it's hitting the board. By 1000 it's consistent.

The scorer is just dartScore - a function that returns points based on where the dart lands relative to the bullseye. Bullseye = 50, inner bull = 25, outer rings scale down from there. Miss the board entirely, score zero. The optimizer doesn't know anything about darts; it just finds parameters that maximize this number.

The arm skeleton

The arm is a kinematic chain: shoulder (fixed) → elbow (dynamic, driven by the throw arc) → wrist → hand → fingers. Everything is Canvas 2D - no physics engine, no WebGL.

The elbow tracks the upper arm rotation as a function of the current forearm angle. I parameterized the upper arm between two extreme angles (wind-up and release) and interpolated:

}

The wrist snap is baked in as an offset to the hand angle:

1
2
const wr_t = Math.max(0, Math.min(1, (ang - WIND2) / (OPT2 - WIND2)));
const hand_ang = ang + (-0.38 + (0.38 + 0.14) * wr_t);

At wind-up the wrist is cocked back 0.38 radians. At release it snaps 0.14 radians forward. Everything distal to the wrist (hand, fingers, dart) uses hand_ang and its perpendicular to compute positions.

Each finger is three phalanges. I store lengths and a CURL interpolation factor (0 = open hand, 1 = dart grip). During the throw the grip stays at 1.0. At release it fades to 0 over five frames as the fingers open.

const CURL_FULL = [1.00, 0.85, 0.35, 0.18]; // index to pinky
const CURL = CURL_FULL.map(c => c * grip + 0.04 * (1 - grip));

function bone(x1, y1, x2, y2, r1, r2, fill) {
  const dx = x2 - x1, dy = y2 - y1, len = Math.sqrt(dx*dx + dy*dy) || 1;
  const nx = -dy / len, ny = dx / len;
  const mx = (x1 + x2) / 2, my = (y1 + y2) / 2;
  const rm = (r1 + r2) * 0.28;
  ctx.beginPath();
  ctx.moveTo(x1 + nx*r1, y1 + ny*r1);
  ctx.quadraticCurveTo(mx + nx*rm, my + ny*rm, x2 + nx*r2, y2 + ny*r2);
  ctx.lineTo(x2 - nx*r2, y2 - ny*r2);
  ctx.quadraticCurveTo(mx - nx*rm, my - ny*rm, x1 - nx*r1, y1 - ny*r1);
  ctx.closePath();
  ctx.fillStyle = fill; ctx.fill();
}

That rm = (r1 + r2) * 0.28 produces a natural waist in the bone shape without needing separate control points.

The dart and its grip

This took more iteration than I'd like to admit. The first version had the dart perfectly aligned with the forearm direction - it just looked like a longer arm. The second version had the grip position at the wrist, which meant the dart floated disconnected from the hand in any non-horizontal arm pose.

The fix: give the dart its own angle (aimed at the board, tilting only slightly with the wrist snap), and anchor the grip to the actual index finger P1 joint rather than an offset from the wrist.

const gx = mc_idx_x + Math.cos(p1A_idx) * 13;
const gy = mc_idx_y + Math.sin(p1A_idx) * 13;

Real dart proportions, for reference: a standard steel-tip dart is about 155mm total - 25mm tip, 50mm barrel, 35mm shaft, 45mm flights. An adult hand from wrist to fingertip is roughly 190mm. So the dart is about 82% of hand length. At the render scale (~1px/mm), the dart should be around 115px. I got this wrong twice before landing on it.

The barrel diameter is about 7mm - roughly 35% of a finger width. Worth knowing because a 13px lineWidth barrel next to 4px finger bones looks wrong; they need to be much closer in proportion.

The dart physics

In the top panel, the dart is a simple projectile:

1
2
3
4
function dartPos2(ang, t, spd = DART_SPD) {
  const vx = Math.cos(ang) * spd, vy = Math.sin(ang) * spd;
  return [LAUNCH_X + vx * t, LAUNCH_Y + vy * t + 0.5 * GRAV * t * t];
}

The dart rotates as it falls, which is just Math.atan2(vy + GRAV * t, vx) - pointing slightly nose-down as gravity pulls it. At release, a 3-frame crossfade transitions from the in-hand dart (bottom panel skeleton) to the flying dart (top panel physics). It's a small thing but without it the throw looks like the dart teleports.

What I'd do differently

The CEM converged to throwing faster than intended (speed ~34 vs the initial target of 18) because at higher speeds the trajectory is flatter and easier to aim. I'd add a penalty term for speed deviation from a "natural" range - the physics are correct but a 34px/frame dart looks unrealistically fast compared to a real throw.

The wrist snap as a pure angle bias has a degeneracy problem: the optimizer can achieve the same bullseye result with different combinations of ang and snapBias. In practice it still converges to a consistent solution, but a more physically grounded model would couple wrist snap to both angular offset and speed increase simultaneously.
